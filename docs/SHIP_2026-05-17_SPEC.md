# Ship 2026-05-17 — Attendee Lockdown + Creator Simplify + Feed-only Event Sync

Spec source-of-truth for the three-part ship that follows the 2026-05-13 Splitter/CreatorHome roll-out. Tracker: `docs/SHIP_2026-05-17_TRACKER.md`.

**Goal.** Hide attendee-facing features that aren't ready (ticket collection, profiles, MyTickets) without making the app feel broken; make the multi-page Site Builder the single canonical path for creating events; let creators add an event to one of their already-deployed sites in one click without rebuilding the bundle.

**Out of scope.** Etherna gateway integration — *deferred*. We design for it (gateway-agnostic feed writes; one default-gateway dropdown that's WoCo today, Etherna later) but ship nothing Etherna-specific in this round.

### `SoonPill.svelte` brief — **must hit the brand aesthetic, not a generic SaaS badge**

Theme is **Concrete & Acid** (`memory/project_ui_theming_direction.md`): RA.co editorial × retro arcade × graffiti. The pill is a *stencil stamp* / sticker stuck on the icon, **not** a rounded "Coming Soon" tag.

**Spec:**
- Markup: `<span class="soon-pill"><span class="prefix">//</span><span class="word">SOON</span></span>`
- **No rounded pill.** `border-radius: var(--radius-sm)` (2px) max. Sharp rectangular feel.
- Font: `SOON` in **Bungee** (`var(--font-tag)`) — this is one of the rare graffiti-accent moments the palette doc reserves Bungee for. `//` prefix in JetBrains Mono.
- Sizing: `font-size: 9px`, `letter-spacing: 0.08em` on the word; prefix slightly dimmer (`--text-muted`).
- Colour: `SOON` is `var(--accent)` (acid lime `#C7F23A`); chip background `var(--accent-ink)` (`#0B0B09`); 1px hairline border in `var(--accent)` at 40% alpha (`color-mix(in srgb, var(--accent) 40%, transparent)`).
- **Slight skew/rotation:** `transform: rotate(-4deg)` — graffiti tag tilt. Apply `transform-origin: 100% 50%` so it pivots from the icon corner.
- **Sticker-lift outline:** `outline: 2px solid var(--bg-elevated)` so the pill reads as *sat on top of* the icon rather than welded to it. (Matches the bottom-nav background.)
- **Position:** absolute top -4px right -10px relative to the nav icon container, slightly overhanging — sticker on a corner.
- Padding: `2px 5px 1px 4px` (asymmetric, slightly heavier left from the `//`).
- No animation by default. Optional reduced-motion-respecting flicker (one frame every ~4s, opacity 0.85→1) — **skip unless `/frontend-design` strongly wants it**; risk of feeling gimmicky.

**Reference reading:** look at how `.live-dot`, `.kicker-tag`, and the `divider-tag` are styled in `apps/web/src/lib/landing/Splitter.svelte` for the brand voice. **Reuse those tokens.** Do not invent new colours.

### `ComingSoon.svelte` brief (use `/frontend-design` — feed it this whole section)

Same theme rules as the pill. Reuse Splitter and CreatorHome scaffolding tokens.

- **Layout.** Single editorial column, centred, max 560px wide, generous vertical breathing room (top padding ~`4rem`). Mobile-first; pads down to 1.25rem at narrow widths.
- **Top kicker.** Mono dotted `// FEATURE IN BUILD` in `var(--text-muted)`, matching the `.mono-kicker` pattern used in Splitter hero.
- **Headline.** Display weight, large clamp `clamp(2rem, 5vw, 3rem)`, tight letter-spacing (-0.03em). Pulls from a feature map:
  - `tickets` → "Your tickets, in your **pocket**."
  - `profile` → "Your scene, your **face**."
  - The bolded word wraps in `<span class="tag-display">` for the Bungee graffiti accent in `var(--accent)`.
- **Subhead — explain the WHY.** One short paragraph in `var(--text-secondary)`, max ~70ch:
  > "We want attendee accounts to be self-custodied from day one — your tickets and your profile, on your keys. We're finishing the client-side feed signers that make that possible. Until then, every ticket lands in your inbox."
- **Email capture (optional).** Single input + adjacent ghost button "Get notified →". Hairline border, square corners (`var(--radius-sm)`). On submit POST `/api/waitlist` with `{ email, feature }`; if the endpoint isn't yet wired, log + show success toast — gate behind a `VITE_WAITLIST_ENABLED` flag.
- **Secondary action.** Quiet text link below: `← Browse events` → `/discover`.
- **Texture.** Apply `.scanlines` + `.grain` overlay classes (already in `app.css`) at low opacity — same as Splitter hero. Reinforces the retro/concrete feel without dominating.
- **Corner stamp (optional).** Bottom-right corner of the section, tiny mono `BUILD // 2026-05` in `--text-dim`, like the Splitter's `.corner-br` stamp. Hide on mobile.

**Anti-patterns to avoid:** centred soft-glow card, gradient blob backgrounds, generic "Stay tuned!" copy, rounded 9999px chip badges, emoji icons, pastel anything.

## P4 — Feed-only "Add event to my sites"

Server endpoints **already exist** — see `apps/server/src/routes/sites.ts:305` (`POST /api/sites/:id/events`) and `:345` (`DELETE /api/sites/:id/events/:eventId`). Both write only to the `woco/site/{siteId}/events` feed — no BZZ rebuild, no Etherna re-upload, deployed sites pick it up on next visitor read (server-side cache TTL = 5min; can be bypassed with cache bust if needed).

### Decisions

| | Decision |
|---|---|
| Where the "Add to site" UI appears | (a) **In SiteBuilder Step 2** (deploy targets) as an additional toggle list of all the creator's sites. (b) **Standalone surface** at `/creator/sites/:siteId/events` for ongoing management of an already-deployed site. |
| Behaviour at creation time | Multi-select checkboxes of creator's sites. On successful event publish, server is called once per selected site with `POST /api/sites/:id/events`. Failures per-site shown inline; do not block deploy. |
| Behaviour post-deploy | The `/creator/sites/:siteId/events` page lists the site's current events with toggle to add/remove from creator's full event list. Reuses `EventsTab.svelte` logic — extract into a shared `<SiteEventsManager siteId={...} />` component. |

## Etherna read (architecture note, no code change)

User's mental model is correct and the spec assumes it:

1. **WoCo Bee writes feeds.** Site config feed, events index feed, per-event feeds — all signed by `FEED_PRIVATE_KEY` and stamped by `POSTAGE_BATCH_ID` on our server. Feeds are addressed by `(owner, topic)` and resolvable from *any* Swarm gateway.
2. **Etherna will serve the static bundle.** When the deployed site loads on `<sub>.eth.limo` or an Etherna URL, it calls our API → our API reads our feeds via our Bee → response goes back to the browser. The browser does not need to know whose Bee wrote the feed.
3. **Implication for this ship:** the `GatewayPicker` registry's `url` field is *only* the BZZ-upload target for the bundle, not the feed-read endpoint. The feed path stays on our Bee regardless of which gateway is selected. When Etherna lands, we add `{ id: "etherna", url: "https://gateway.etherna.io" }` and route `/api/sites/:id/deploy` to upload there instead of our gateway.
